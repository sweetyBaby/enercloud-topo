# 储能拓扑 · 实时数据接口契约

> 给后端同学对接用。后端要做的事只有一句：**把每个被监控的设备/全局量，拍平成一个 `{ 信号名: 值 }` 的 JSON 对象交给前端**。无嵌套、无固定字段表——键名由「画布里配的元素 id + 字段英文名」决定，值就是当前测量值。
>
> 渲染端参考实现：`topo-editor/topology-editor.js` 中 `applyLiveSignals()`（实时数据入口）、`parseSignal()`（键名解析）、`buildCtx()` / `computeDynamic()`（按信号求显隐/流向）。本页描述的就是这套实现对「实时数据 JSON」的对外契约。

---

## 1. 一句话数据结构

实时数据是**一个扁平 JSON 对象**，键是「信号名」，值是该信号的当前值。**没有任何嵌套层级，没有时间戳/元信息包裹**（如需包裹请在网关层拆开）。

```json
{
  "grid_1.P(kW)": 383,
  "pcs_1.P(kW)": -9,
  "pcs_1.Status": "放电",
  "bms_1.SOC(%)": 55,
  "mode": "island"
}
```

> 注：信号键**只有两类** —— 「节点数据字段」(`节点id.英文字段名`) 和「全局信号」。`status` / `online` 这类**凭空字段已移除**：设备若要表达运行状态/连通性，就在画布里给它**显式加一个数据字段**（中文名 `状态`/`在线`，英文名 `Status`/`Online`），再用 `节点id.英文名`（如 `pcs_1.Status`）下发即可——键名一律来自画布里真实绑定字段的**英文名**，不再有隐藏字段。

> 这同一份对象**同时**承担两件事：① 驱动规则（决定元素显隐、连线流向）；② 显示在元素的字段卡片上。后端**只需推这一份**，无需区分。

---

## 2. 信号名（键）的命名规则

| 信号类型 | 键格式 | 示例 | 值的含义 |
|---|---|---|---|
| **节点数据字段** | `节点id.英文字段名` | `pcs_1.P(kW)`、`bms_1.SOC(%)`、`pcs_1.Status` | 该设备该字段的实测值；**所有可下发的节点信号都来自画布里绑定的数据字段** |
| **全局信号** | `信号名`（无前缀点号或前缀非节点 id） | `mode`、`islanding` | 全局运行量，任意规则可引用 |

> ⚠️ 旧版的 `节点id.status` / `节点id.online` 这两个**自动隐藏字段已取消**。运行状态、是否在线等，现在都必须是节点上**显式绑定的数据字段**（中文名 `状态`/`在线`，英文名 `Status`/`Online`），键名用 `节点id.英文名`。这样后端要推哪些键，完全可由画布 JSON 的 `nodes[].data[].key.en` 枚举得到，不存在「画布里看不到、却要推」的隐藏键。

**关键约定：**

- **`节点id`** = 运营端给元素配置的 ID（如 `pcs_1`）。配好后**尽量不要改**，它是所有信号键的前缀。
- **`字段名`用运营端配置的「英文名」**（`data[].key.en`），如 `P(kW)`、`SOC(%)`、`Today(kWh)`、`Status`。运营端每个字段的中文名、英文名**都是必填**（缺任一会阻断导出），英文名作端到端信号键，前后端必须逐字一致（含大小写、括号）。图上字段卡片仍显示中文名，仅信号键用英文。
- **键名解析逻辑**（`parseSignal`）：取键名里**最后一个 `.`**，若其左侧字符串**匹配画布中存在的某个节点 id**，则解析为「节点信号」，否则整串当作**全局信号名**。
  - 因此字段名内部即便没有点也安全；而 `mode`（无点）自然落为全局信号。
  - 推论：若某个 `节点id.字段` 的节点 id 在画布里不存在，会被静默当成全局信号、不会报错，但也不会更新对应卡片——**对不上的键名等于无效**，务必与画布 id 对齐。

## 3. 值的类型与显示语义

| 值 | 规则求值 | 卡片显示 |
|---|---|---|
| 数字（`383`、`-9`、`0`） | 参与数值比较 | 如实显示，**`0` 显示为 `0`**（不是空） |
| 字符串（`"放电"`、`"island"`） | 参与 `==`/`in`/`between` 等 | 如实显示 |
| 布尔（`true`/`false`） | `truthy`/`falsy`/`==` 等 | 按字符串显示（不想显示就把该字段设为隐藏） |
| `null` / 空串 `""` / **不传该键** | 该信号视为「无值」，数值比较一律**不命中**（"没有数据 ≠ 0"） | 显示为空（`字段名: `） |

> **想表达"暂无数据"**：给该键传 `null`/`""`，或干脆不传。**要清空已显示的值必须显式传空**——见下条增量语义。

## 4. 三种下发方式（后端按部署形态选其一）

读取端有三条独立链路，**数据结构完全相同**，区别只在「怎么把这份 JSON 送进渲染器」。

### 方式 A · URL 轮询（最简单，后端只需提供一个 GET 接口）

前端按 `interval` 毫秒轮询，每次 `GET` 拿**完整的一份**当前快照（`cache:'no-store'`）。

```
拓扑结构编辑器-V3.html?mode=runtime&topology=<画布JSON地址>&signals=<实时数据接口URL>&interval=2000
```

| 后端需保证 | 说明 |
|---|---|
| 返回 JSON | `Content-Type: application/json`，UTF-8（字段名含中文） |
| 每次返回**全量快照** | 轮询是「整批覆盖」，每次返回当前所有信号的最新值 |
| 允许跨域 | 前端走 `fetch`，跨域需配 CORS（或改用方式 B 规避） |

### 方式 B · iframe + postMessage（大屏/管理后台推荐，可推送、免 CORS）

父页面把数据 `postMessage` 给 iframe。**收到 `topo:ready` 后再推**。

| type | data | 语义 |
|---|---|---|
| `topo:signals` | `{信号:值}` | **整批覆盖**：清空旧的规则上下文后整体重置（首帧/重置用） |
| `topo:merge` | `{信号:值}` | **增量合并**：只更新传入的键，其它保留上次值（**高频推荐**） |

```js
// 首帧整批
frame.contentWindow.postMessage({ type:'topo:signals',
  data:{ 'grid_1.P(kW)':383, 'pcs_1.P(kW)':-9, 'bms_1.SOC(%)':55, 'pcs_1.Status':'放电' } }, '*');
// 之后每次只推变化的信号
frame.contentWindow.postMessage({ type:'topo:merge', data:{ 'pcs_1.P(kW)':-12 } }, '*');
```

### 方式 C · JS API（同源/直接托管）

```js
TopoRuntime.setSignals({...});    // 整批覆盖（= topo:signals）
TopoRuntime.mergeSignals({...});  // 增量合并（= topo:merge）
```

---

## 5. 覆盖 vs 合并 —— 后端必须理解的语义

| 行为 | 整批覆盖（轮询 / `topo:signals` / `setSignals`） | 增量合并（`topo:merge` / `mergeSignals`） |
|---|---|---|
| 规则上下文 | **先清空**再写入本批 | 在上次基础上叠加本批 |
| 本批**未包含**的信号 | 规则求值回退到**画布里的静态默认值**（节点字段初值 / 全局信号样例） | 保留上次推送的值 |
| 适用场景 | 首帧、状态重置、低频全量 | 高频更新、只有少量信号变化 |

> 经验做法：**首帧用整批**建立完整基线，**之后用 `topo:merge` 只推变化量**。轮询方式 A 因为是无状态拉取，每次都按整批覆盖处理，所以**每次必须返回全量**，否则上次的值会回退到静态默认。

## 6. 一份贴近真实的示例

某园区储能：电网、光伏、PCS、电池、负载，外加全局运行模式。

> 前提：下面用到的 `Status`（状态）、`Online`（在线）等都是这些设备在画布里**已绑定的数据字段**（键名用字段英文名，不是隐藏字段）。

```json
{
  "grid_1.P(kW)": 383,
  "grid_1.Online": true,
  "grid_1.Status": "并网",

  "pv_1.P(kW)": 126,
  "pv_1.Today(kWh)": 540,

  "pcs_1.P(kW)": -9,
  "pcs_1.Status": "放电",

  "bms_1.SOC(%)": 55,
  "bms_1.Status": "正常",

  "load_1.P(kW)": 500,

  "mode": "island",
  "islanding": false
}
```

- `grid_1.Online:true` → 若「电网」配了 `visibleWhen` 引用该字段（英文名 `Online`），则据此控制可见（前提：电网节点绑定了「在线」数据字段）。
- `pcs_1.P(kW):-9` → 配套的 `dirRules`（正充负放）把 PCS↔电池连线流向判为「放电方向」。
- `bms_1.Status:"放电"` → 若电池 BMS 配了 `iconRules`（条件引用英文名 `Status`），据此把图标切到 `bms_discharge`（方案 A 同一渲染器自动换图；方案 B 读 `resolveDynamic` 返回的 `node.iconType`，用 `ICON_PATHS[iconType]` 取图）。
- `bms_1.SOC(%):55` → 显示在电池卡片上；同时可被告警规则（如 `< 20`）引用。
- `mode:"island"` → 全局信号，可驱动「离网时才显示某些连线」等规则。

## 7. 后端对接清单

| 项 | 要求 |
|---|---|
| **键名对齐** | 与画布 JSON 里的 `节点id` 和**字段英文名**逐字一致（运营端导出的画布 JSON 即真值来源；可读 `nodes[].id` 与 `nodes[].data[].key.en`） |
| **扁平结构** | 顶层就是 `{信号:值}`，禁止嵌套/数组/包裹层 |
| **编码与类型** | UTF-8 JSON；数值给 number 而非字符串（虽 `0=="0"` 可互通，但数值比较需真数字） |
| **全量 vs 增量** | 轮询每次返回全量；postMessage 高频场景用 `topo:merge` 只推变化量 |
| **空值表达** | 无数据传 `null`/`""` 或不传；`0` 是真实值不要用它表示"无" |
| **跨域** | 方式 A 需 CORS；不便配置时用方式 B（iframe postMessage） |
| **频率** | 渲染按动画帧自动重绘，后端按业务需要的刷新率推送即可（常见 1–5s） |

> 信号名「有哪些」由当前画布决定。后端可在联调前向运营端要一份导出的**画布 JSON**，据 `nodes[].id` + `nodes[].data[].key` + 顶层 `signals[].name` 枚举出全部应推送的键。
