# 储能拓扑编辑器 PRD

版本：v1.0  
日期：2026-07-01  
适用项目：`topo` 储能拓扑编辑器与运行态渲染器

## 1. 产品概述

### 1.1 背景

储能、光伏、负载、PCS、BMS、EMS 等设备在运营平台中需要以拓扑图方式展示设备关系、能量流向、实时状态和关键指标。传统静态图无法满足运行状态动态变化、设备字段绑定、前端大屏复用和模板复用的需求。

本项目提供一套储能拓扑编辑与运行展示能力：运营端通过可视化编辑器配置拓扑、数据字段、后台绑定、动态规则和模板；前端运行态复用同一份渲染器，根据拓扑 JSON 和实时信号渲染只读运行画面。

### 1.2 产品定位

本产品是面向储能/能源管理系统的可视化拓扑配置工具，核心定位为：

- 运营端低代码配置拓扑图。
- 前端运行态像素级复用编辑器渲染能力。
- 后端只需按约定提供扁平实时信号。
- 模板、图标、字段字典、数据绑定可持续沉淀。

### 1.3 一句话目标

让运营人员无需写代码即可搭建储能系统拓扑图，并让前端通过同一份拓扑 JSON 和实时数据自动展示设备显隐、图标变化、字段值和流向动画。

## 2. 用户与场景

### 2.1 用户角色

| 角色 | 主要诉求 |
|---|---|
| 运营配置人员 | 拖拽搭建拓扑、配置设备字段、设置动态规则、保存模板 |
| 前端开发人员 | 复用编辑器运行态，快速接入拓扑 JSON 与实时数据 |
| 后端开发人员 | 根据画布导出的字段绑定清单提供实时数据接口 |
| 项目实施人员 | 针对不同站点快速复用模板并调整设备关系 |
| 运维/大屏用户 | 查看实时拓扑、设备状态、能量流向和关键指标 |

### 2.2 核心使用场景

1. 运营人员创建一个园区储能拓扑图，包含电网、光伏、PCS、电池、负载、EMS 等元素。
2. 运营人员为每个元素配置名称、图标、数据字段、后台设备实例和字段来源。
3. 运营人员配置规则，例如 SOC 低于阈值隐藏某些连线、PCS 功率正负决定能量流向、BMS 状态决定图标切换。
4. 运营人员使用注入信号在编辑器中验证规则效果。
5. 运营人员导出画布 JSON 或保存为模板。
6. 前端以 runtime/embed 模式加载同一页面，并通过 URL 轮询、postMessage 或 JS API 下发实时数据。
7. 运行态根据实时信号自动更新字段值、显隐、图标和流向。

## 3. 产品目标

### 3.1 业务目标

- 降低储能拓扑图配置成本，减少前端重复开发。
- 提升不同项目拓扑复用能力，支持模板沉淀。
- 统一运营端编辑效果与前端运行展示效果。
- 让后端实时数据接口保持简单、可枚举、可联调。

### 3.2 用户目标

- 配置人员可以通过拖拽完成拓扑搭建。
- 配置人员可以直观看到规则是否生效。
- 前端可以最小成本嵌入拓扑运行态。
- 后端可以从导出的 `dataBindings` 和 `nodes[].data[]` 明确知道要推哪些信号。

### 3.3 成功指标

| 指标 | 目标 |
|---|---|
| 新建常规储能拓扑耗时 | 小于 30 分钟 |
| 前端接入运行态耗时 | 小于 1 天 |
| 规则配置后本地验证闭环 | 支持编辑器内即时预览 |
| 模板复用 | 支持保存、编辑、重命名、删除、按需加载 |
| 数据接口复杂度 | 实时数据为单层扁平 JSON |

## 5. 功能需求

### 5.1 拓扑编辑画布

#### 需求描述

系统应提供一个可视化画布，支持从左侧元素库拖拽设备到画布，编辑节点、连线、字段和布局。

#### 功能点

- 支持节点拖拽创建。
- 支持画布平移、缩放、缩放复位。
- 支持节点拖动、框选、多选、复制、删除。
- 支持撤销、重做。
- 支持清空画布。
- 支持左侧元素面板和右侧属性面板折叠。
- 支持画布背景、主题、全局字体样式配置。
- 支持网格、数据字段、连线标签、占位点标记的显示开关。

#### 验收标准

- 用户可以从元素库拖入节点并在画布中移动。
- 用户可以通过鼠标滚轮缩放画布。
- 用户可以撤销和重做主要编辑操作。
- 选中节点或连线后，右侧属性面板能展示对应属性。

### 5.2 元素库与图标库

#### 需求描述

系统应内置储能拓扑常用元素，并支持图标库扫描和自定义图标上传。

#### 元素分类

- 电源侧：主电网、光伏板、发电机。
- 储能设备：PCS、电池 BMS、电池柜、氢储能。
- 母线/主干线：母线、交流主干线、直流主干线、联络线。
- 电气设备：变压器、断路器、高压箱。
- 计量与负载：计量表、关口表、用电负载、充电桩。
- 开关元件：断路器、隔离开关、刀闸、接触器、熔断器等。
- 辅助系统：EMS、空调、消防、传感器。
- 无源元件：电阻、电感、电容、CT、PT、SPD、接地。
- 辅助元素：占位点、文本框、变量。

#### 功能点

- 左侧元素库按分组展示。
- 支持搜索元素。
- 支持分组展开/折叠。
- 支持重新扫描图标库。
- 支持上传 PNG、SVG、JPEG 自定义图标。
- 支持导出完整元素库包 ZIP。

#### 验收标准

- `icons/index.json` 中登记的元素能出现在左侧元素库。
- 未登记但存在于 `icons/` 的图标可归入自定义图标分组。
- 自定义图标添加后可拖入画布并参与导出。

### 5.3 节点属性配置

#### 需求描述

用户应能配置节点的身份、展示、样式、数据字段、后台绑定和运行态交互。

#### 功能点

- 配置节点 ID。
- 配置中文标签和英文标签。
- 配置节点类型。
- 配置图标大小、旋转角度、标签字号、标签颜色。
- 配置是否显示名称、是否显示数据字段。
- 配置文本框背景、边框、圆角。
- 配置变量元素的标签和值样式。
- 配置占位点填充色和透明度。
- 配置运行态事件绑定：左键点击、右键点击、双击，跳转当前页或新窗口。
- 展示节点坐标。

#### 规则与限制

- 节点 ID 是实时信号键和 `dataBindings` 的主键来源，应保持唯一。
- 节点 ID 为空或重复时，应阻断导出。
- 中文标签用于中文界面展示，英文标签用于英文界面展示。

#### 验收标准

- 修改节点属性后画布即时更新。
- 节点 ID 重复或为空时，导出被阻断并提示用户修正。
- 运行态点击事件只在预览/运行态触发。

### 5.4 数据字段配置

#### 需求描述

用户应能为节点添加数据字段，并配置字段中英文名、默认值、显示位置和后台字段绑定。

#### 功能点

- 添加、删除数据字段。
- 配置字段中文名和英文名。
- 配置字段默认值。
- 拖动字段卡片位置。
- 批量显示或隐藏所选元素字段。
- 字段可绑定后台设备字段。
- 字段绑定支持跟随节点设备，也支持跨设备显式来源。

#### 规则与限制

- 字段中文名、英文名均必填。
- 同一节点内字段中文名和英文名不能重复。
- 字段英文名是实时数据键的字段段，例如 `pcs_1.P(kW)` 中的 `P(kW)`。
- 字段命名问题应阻断导出。
- 未绑定后台字段允许导出，但应给出风险提示。

#### 验收标准

- 字段名缺失或重复时，界面能高亮并阻断导出。
- 已绑定字段在导出 JSON 的 `dataBindings` 中生成对应记录。
- 实时数据命中字段键时，字段卡片显示值更新。

### 5.5 后台设备绑定

#### 需求描述

系统应支持从后台设备类型、设备实例、字段字典中选择字段来源，形成可供后端对接的数据绑定清单。

#### 数据来源

- `device/device-type.json`：设备类型。
- `device/device-info.json`：设备实例。
- `dic/*.json`：设备字段字典。
- `dic/index.json`：运行时扫描合并后的字段字典清单。

#### 功能点

- 节点选择默认后台设备类型和设备实例。
- 字段绑定选择设备实例、字段分类 location 和字段 field。
- 全局信号绑定必须选择具体设备实例和字段。
- 支持刷新后台设备/字典数据。
- 导出时生成 `dataBindings`。

#### 验收标准

- 用户可以为节点选择设备实例。
- 用户可以为字段选择后台字段。
- 导出 JSON 包含完整的 `dataBindings`：`signal`、`node`、`label`、`source.deviceType`、`source.deviceId`、`source.field`。
- 设备实例或字段字典失效时，导出风险区应提示。

### 5.6 连线配置

#### 需求描述

系统应支持节点间连线，并配置连线类型、颜色、样式、走线方式、流向、标签和动态规则。

#### 功能点

- 支持连线模式和连续连线。
- 支持连线类型选择。
- 支持全局线宽和单条线宽。
- 支持连线颜色跟随类型或自定义。
- 支持线条样式：实线、虚线、点线等。
- 支持走线方式：智能、直线、弧线、手动拐点。
- 支持拖动拐点并吸附对齐。
- 支持固定流向：正向、反向、双向、无流向。
- 支持单独显示连线标签。
- 支持重置为智能走线或直线走线。

#### 验收标准

- 用户可连接两个节点并形成可见连线。
- 选中连线后可以修改连线类型和流向。
- 智能走线应尽量避开节点并减少交叉。
- 手动拐点可被拖动调整。

### 5.7 自动布局与对齐

#### 需求描述

系统应提供自动布局和多选对齐能力，帮助用户快速整理拓扑。

#### 功能点

- 一键自动布局。
- 智能整理走线。
- 多选节点左对齐、右对齐、顶部对齐、底部对齐、水平居中、垂直居中。
- 水平/垂直等距分布。
- 按指定间距排列。
- 排成一行、一列、矩阵。
- 居中于画布。
- 支持字段卡片多选对齐。

#### 验收标准

- 多选两个及以上节点时显示对齐工具栏。
- 点击对齐按钮后，所选节点坐标按预期变化。
- 自动布局后节点和连线应保持可读。

### 5.8 母线与主干线展示

#### 需求描述

系统应支持储能拓扑中常见的母线汇流视觉效果，使多条连线可汇聚为主干线。

#### 功能点

- 支持母线汇流排、加粗实线、双线母线、发光母线样式。
- 支持主干线加粗显示。
- 支持跨设备共享主干。
- 支持主干通道间距调整。
- 支持拖动主干线中点手柄微调位置。
- 支持重置所有主干位置。

#### 验收标准

- 多条相近连线可合并显示为主干线。
- 用户可以调整主干线样式和间距。
- 主干线位置重置后恢复自动计算效果。

### 5.9 数据驱动规则

#### 需求描述

系统应支持声明式规则，根据实时信号动态控制节点显隐、节点图标、连线显隐和连线流向。

#### 规则类型

| 规则 | 作用对象 | 字段 |
|---|---|---|
| 节点显示条件 | 节点 | `visibleWhen` |
| 节点图标规则 | 节点 | `iconRules` |
| 连线显示条件 | 连线 | `showWhen` |
| 连线流向规则 | 连线 | `dirRules` |

#### 条件能力

- 支持 `all`、`any`、`not` 条件组合。
- 支持 `==`、`!=`、`>`、`>=`、`<`、`<=`、`in`、`between`、`truthy`、`falsy`、`exists`。
- 支持与常量比较。
- 支持与另一个信号引用比较。
- 未配置条件时默认恒为真。

#### 执行规则

- 节点显示条件不满足时，运行态隐藏该节点。
- 节点隐藏后，与其相连的连线也不展示。
- 节点图标规则按顺序匹配，首个命中生效，不命中回落到节点自身图标。
- 连线显示条件不满足时不绘制该连线。
- 连线流向规则按顺序匹配，首个命中生效，不命中回落到固定流向。

#### 验收标准

- 规则保存后在编辑器中实时生效。
- 编辑态下被隐藏元素应虚化显示并仍可编辑。
- 运行视图下被隐藏元素应彻底隐藏。
- 规则总览能列出全部已配置规则。

### 5.10 信号管理与注入测试

#### 需求描述

系统应提供信号面板，用于管理全局信号、注入测试数据和批量样例 JSON。

#### 功能点

- 添加全局信号。
- 配置全局信号中文名、英文名和默认值。
- 全局信号可绑定后台字段。
- 支持按元素/全局信号添加注入行。
- 支持注入值输入。
- 支持注入分组展开/折叠。
- 支持生成当前信号模板 JSON。
- 支持粘贴 JSON 批量应用注入值。
- 支持清空注入。

#### 规则与限制

- 全局信号中文名和英文名均必填。
- 全局信号中文名和英文名需全局唯一。
- 全局信号英文名作为实时数据键，例如 `mode`。
- 全局信号命名问题应阻断导出。
- 注入数据只用于编辑器测试，不作为正式实时数据接口。

#### 验收标准

- 用户可新增全局信号并在规则中引用。
- 注入信号后，画布规则和字段展示即时更新。
- 全局信号缺名或重复时，导出被阻断。

### 5.11 画布 JSON 导入导出

#### 需求描述

系统应支持导入和导出完整画布 JSON，作为运营配置与前端运行态之间的核心契约。

#### 导出内容

- `schemaVersion`。
- `meta.libraryRef`。
- `meta.canvas`。
- `meta.view`。
- `edgeStyles`。
- `nodes`。
- `edges`。
- `signals`。
- `sampleSignals`。
- `dataBindings`。

#### 功能点

- 显示画布 JSON。
- 复制 JSON。
- 下载 JSON。
- 导入 JSON 并还原画布。
- 导出前校验节点 ID、字段名、全局信号名。
- 导出风险提示未绑定字段、失效设备实例、失效字典字段。

#### 验收标准

- 导出的 JSON 可被再次导入并还原主要画布内容。
- 导出的 JSON 可被运行态加载。
- 严重数据问题会阻断导出。
- 非阻断风险会提示但允许导出。

### 5.12 模板库

#### 需求描述

系统应支持将当前画布保存为模板，并支持模板浏览、加载、编辑、重命名和删除。

#### 功能点

- 打开模板库。
- 展示模板名称、英文名、描述和缩略图。
- 加载模板到画布。
- 保存当前画布为模板。
- 编辑已有模板内容。
- 重命名模板。
- 删除模板。
- 支持默认模板。
- 模板清单按需加载，选中模板后再加载完整模板 JSON。

#### 数据模型

- `templates/index.json`：模板清单。
- `templates/<id>.json`：单个模板。
- 模板文件支持 `seed` 或 `canvas` 两种形态。

#### 接口

- `GET /api/templates`：取模板清单。
- `POST /api/templates`：新建模板。
- `PUT /api/templates/:id`：重命名或编辑模板。
- `DELETE /api/templates/:id`：删除模板。

#### 验收标准

- 本地开发和生产 Node 服务下，模板写入能落盘到 `templates/`。
- 纯静态托管下，模板可读；写入失败时回退为下载 JSON。
- 删除默认模板后，默认模板应自动切换到剩余模板。

### 5.13 运行态渲染

#### 需求描述

系统应支持只读运行模式，使前端或大屏可直接复用编辑器渲染能力。

#### 接入方式

1. URL 参数：

```text
topo.html?mode=runtime&topology=<画布JSON地址>&signals=<实时数据地址>&interval=2000
```

2. iframe + postMessage：

```js
frame.contentWindow.postMessage({ type: 'topo:topology', data: topology }, '*');
frame.contentWindow.postMessage({ type: 'topo:signals', data: signals }, '*');
frame.contentWindow.postMessage({ type: 'topo:merge', data: partialSignals }, '*');
```

3. JS API：

```js
TopoRuntime.loadTopology(topologyOrUrl);
TopoRuntime.setSignals(signals);
TopoRuntime.mergeSignals(partialSignals);
TopoRuntime.fit();
```

#### 功能点

- 运行态隐藏编辑器外壳。
- 支持加载拓扑 JSON。
- 支持整批覆盖实时信号。
- 支持增量合并实时信号。
- 支持自动适配容器。
- 支持可选平移缩放。
- 支持向父页面发送 `topo:ready`。

#### 验收标准

- 使用 `mode=runtime` 时页面只显示拓扑画布。
- 前端下发拓扑 JSON 后能完成渲染。
- 下发实时信号后，字段值和规则效果能更新。
- `topo:merge` 只更新传入信号，保留其它信号上次值。

### 5.14 实时数据接口

#### 需求描述

后端应提供扁平 JSON 对象作为实时数据，键为信号名，值为当前测量值。

#### 数据格式

```json
{
  "grid_1.P(kW)": 383,
  "pcs_1.P(kW)": -9,
  "pcs_1.Status": "放电",
  "bms_1.SOC(%)": 55,
  "mode": "island"
}
```

#### 信号命名

| 信号类型 | 命名规则 | 示例 |
|---|---|---|
| 节点字段 | `节点id.字段英文名` | `pcs_1.P(kW)` |
| 全局信号 | `全局信号英文名` | `mode` |

#### 值语义

- 数字、字符串、布尔值按原值参与规则和展示。
- `0` 是真实值，应显示为 `0`。
- `null`、空串、未提供表示无值。
- 数值比较遇到无值不命中。

#### 覆盖与合并

- URL 轮询、`topo:signals`、`setSignals` 为整批覆盖。
- `topo:merge`、`mergeSignals` 为增量合并。
- 轮询方式每次必须返回全量快照。

#### 验收标准

- 后端返回顶层扁平 JSON，不嵌套、不包裹元信息。
- 键名与画布 JSON 中的节点 ID 和字段英文名逐字一致。
- 错误键名不会更新目标字段，应在联调时通过画布 JSON 枚举核对。

### 5.15 多语言

#### 需求描述

系统应支持中文和英文界面展示，并支持节点、字段、模板、元素名称的中英文配置。

#### 功能点

- 工具栏支持中英文切换。
- 元素库支持中文标签和英文标签。
- 节点支持中文标签和英文标签。
- 数据字段支持中文名和英文名。
- 全局信号支持中文名和英文名。
- 模板支持中文名和英文名。

#### 验收标准

- 切换语言后，已有节点和字段展示应切换到对应语言。
- 字段英文名仍作为信号键，不因界面语言变化而改变。

## 6. 数据需求

### 6.1 核心实体

| 实体 | 说明 |
|---|---|
| Element / Node | 画布上的设备、文本、变量、占位点 |
| Edge | 节点之间的连线 |
| Data Field | 节点上的实时数据字段 |
| Global Signal | 不属于某个节点的全局信号 |
| Rule | 显隐、图标、流向的条件规则 |
| Template | 可复用拓扑模板 |
| Data Binding | 信号与后台设备字段的映射 |
| Icon Library | 元素类型与图标文件的映射 |

### 6.2 画布 JSON 要求

- 必须包含 `schemaVersion`。
- 节点 ID 必须唯一。
- 节点字段英文名在同节点内必须唯一。
- 全局信号英文名必须全局唯一。
- 可选规则必须为声明式 JSON，不允许执行任意代码。
- 导出 JSON 应自包含运行态需要的视图设置、连线样式和规则配置。

### 6.3 数据绑定要求

`dataBindings` 应能支持后端枚举数据源，单条绑定结构应包含：

```json
{
  "signal": "bms_1.SOC(%)",
  "node": "bms_1",
  "label": "SOC(%)",
  "source": {
    "deviceType": "BCU",
    "deviceId": "xxx",
    "field": "StringDataLogs.SOC"
  }
}
```

全局信号的 `node` 为 `null`。

## 7. 非功能需求

### 7.1 易用性

- 常用操作应可通过工具栏完成。
- 关键编辑动作应即时反馈到画布。
- 导出前的阻断问题和风险问题应清晰提示。
- 规则配置应能通过注入测试完成闭环验证。

### 7.2 性能

- 常规拓扑图应保持编辑交互流畅。
- 实时数据更新不应要求手动刷新页面。
- 图标库、模板库应按需加载或使用轻量清单。
- 运行态应按动画帧自动重绘。

### 7.3 安全性

- 动态规则必须是声明式 JSON，不使用 `eval`。
- 模板 API 的 `id` 需进行文件名安全过滤。
- 模板请求体需限制大小，避免超大请求拖垮服务。
- 纯静态部署下不应尝试在浏览器本地写服务器文件。

### 7.4 兼容性

- 支持现代桌面浏览器。
- 支持本地开发、生产 Node 服务和纯静态托管三种部署方式。
- 支持 iframe 内嵌。
- URL 轮询方式需考虑 CORS。

### 7.5 可维护性

- 图标清单、设备类型、设备实例、字段字典应文件化管理。
- 模板应单文件存储，清单可扫描生成。
- 编辑器与运行态尽量复用同一套渲染逻辑。
- 后续建议将大型 `topology-editor.js` 按配置、核心状态、渲染、交互、规则、模板、导出等模块拆分。

## 8. 部署需求

### 8.1 本地开发

命令：

```bash
npm run dev
```

能力：

- 托管源码。
- 自动扫描图标、字典和模板。
- 支持模板写入。
- 支持开发热重载。

### 8.2 生产可写部署

命令：

```bash
npm run build
npm start
```

能力：

- 托管 `dist/` 静态资源。
- 模板读写指向项目根 `templates/` 或 `TEMPLATES_DIR`。
- 模板持久化不受重新 build 影响。

### 8.3 纯静态部署

能力：

- 可读取模板和静态资源。
- 不支持模板写入。
- 保存模板失败时回退为下载 JSON。

## 9. 验收清单

### 9.1 编辑器验收

- 能打开 `topo.html` 并正常加载元素库。
- 能拖拽创建节点。
- 能创建连线并设置类型、样式、流向。
- 能配置节点字段和字段绑定。
- 能配置节点显隐规则、图标规则、连线显隐规则、流向规则。
- 能通过注入信号验证规则效果。
- 能导出画布 JSON。
- 能导入画布 JSON 并还原。
- 能保存、编辑、重命名、删除模板。

### 9.2 运行态验收

- `mode=runtime` 能隐藏编辑器外壳。
- 能加载拓扑 JSON。
- 能接收实时数据并更新字段值。
- 节点显隐、图标切换、连线显隐、流向动画与编辑器预览一致。
- iframe 模式能收到 `topo:ready` 并通过 postMessage 下发数据。

### 9.3 后端联调验收

- 后端能根据画布 JSON 枚举所有信号键。
- 实时数据返回顶层扁平 JSON。
- 轮询模式返回全量信号。
- 增量模式只推变化信号。
- `0`、`null`、空串的语义与文档一致。

## 10. 风险与待确认

| 风险/问题 | 说明 | 建议 |
|---|---|---|
| 单文件 JS 过大 | `topology-editor.js` 聚合了编辑、渲染、规则、模板、运行态等职责 | 后续按模块拆分，优先拆纯逻辑 |
| 规则与 runtime 逻辑重复 | 导出包中存在运行端规则逻辑，需避免与编辑器规则引擎漂移 | 抽成共享规则模块 |
| 后端字段命名不一致 | 信号键依赖节点 ID 和字段英文名，大小写/符号不一致会失效 | 联调前以后端读取画布 JSON 枚举键名 |
| 模板写入依赖服务端 | 纯静态部署无法写模板 | 平台后端实现 `/api/templates` 或使用 Node 服务 |
| 复杂拓扑性能 | 大量节点和连线可能影响智能走线与渲染性能 | 后续建立性能基准和大图压测 |
| 权限缺失 | 当前模板 API 无鉴权设计 | 集成到父平台时由父平台网关控制权限 |
